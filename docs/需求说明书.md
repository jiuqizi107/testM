# 测试管理系统需求说明书

---

## 1. 项目概述

### 1.1 项目背景
本系统为银行信息科技部测试室使用的测试管理平台，用于管理测试全生命周期，包括项目、需求、版本、测试用例、缺陷等核心业务。

### 1.2 系统目标
- 实现测试工作全流程管理
- 与OA系统对接，实现版本和需求流转
- 支持多角色权限管理
- 提供数据统计与可视化
- 实现测试资产的沉淀与管理
- AI智能辅助功能

#### 2.1.2 项目详情视图

**项目状态流转规则**：

```
需求分析 → 测试用例编写 → 测试用例已评审 → 集成测试 → 系统测试 → 业务验收测试 → 已投产 → 完成复盘
  ↑           ↑              ↑                 ↑           ↑              ↑           ↑
  └───────────┴──────────────┴─────────────────┴────────────┴──────────────┴──────────┘
                                    可回退至任意前置状态（需记录变更原因）
```

| 流转类型 | 说明 |
|----------|------|
| 正向流转 | 按顺序依次推进，每个状态可自由进入下一状态 |
| 逆向流转 | 可回退至任意前置状态，系统记录：变更前状态、变更后状态、变更原因、变更人、变更时间 |
| 不可跳级 | 禁止跳过中间状态（如从"需求分析"直接跳到"系统测试"） |

```

  │         │        │         │
  │         └────────┴─────────┘（退回后回退至已分配）

  └───────────────────────（可回退至任意前置状态）
```

| 流转类型 | 说明 | 前置条件 |
|----------|------|----------|

| 测试完成 → 已验收 | 验收通过 | 验收人确认 |
| 测试中/测试完成 → 已分配 | 版本退回 | 填写退回原因，触发OA接口调用，自动回退至"已分配" |
| 已验收 | 不可逆状态 | 验收后不可变更 |

| 字段 | 必填 | 说明 |

| 严重程度 | 是 | 致命/严重/一般/轻微 |
| 优先级 | 是 | P1/P2/P3/P4 |
| 缺陷状态 | 是 | 待处理/处理中/已验证/已关闭 |
| 指派人员 | 否 | 分配给指定人员处理 |
| 创建人 | 是 | 自动记录 |
| 创建时间 | 是 | 自动记录 |
| 备注 | 否 | 补充说明 |

**缺陷状态码值**：
- 待处理：新建未分配
- 处理中：已分配正在修复
- 已验证：修复完成待验证
- 已关闭：验证通过关闭

**缺陷状态流转规则**：

```
待处理 → 处理中 → 已验证 → 已关闭
  ↑         ↑        ↑        ↑
  └─────────┴────────┴────────┘
              （可回退或重新打开）
```

| 流转类型 | 说明 |
|----------|------|
| 待处理 → 处理中 | 指派给开发人员处理 |
| 处理中 → 已验证 | 开发人员修复完成，提交验证 |
| 已验证 → 已关闭 | 验证通过，缺陷关闭 |
| 已验证 → 处理中 | 验证失败，退回重新修复 |
| 已关闭 → 处理中 | 重新打开（缺陷复发或验证误判） |
| 禁止跳过 | 禁止从"待处理"直接跳到"已关闭"，必须经过"处理中"和"已验证" |

## 5. 接口设计

### 5.0 通用接口规范

#### 接口幂等性规则

| 接口类型 | 幂等性策略 |
|----------|------------|
| POST /api/projects | 项目编号唯一性校验，重复提交返回409冲突 |
| POST /api/versions | 版本编号唯一性校验，重复创建返回已存在版本 |
| POST /api/requirements | 需求编号唯一性校验，重复创建返回409冲突 |
| POST /api/oa/receive-version | 接收OA幂等token，重复推送返回已存在版本 |
| POST /api/oa/return-version | 退回操作支持重试，失败时进入重试队列 |

#### 错误响应格式

```json
{
  "code": "错误码",
  "message": "错误描述",
  "detail": "详细信息（可选）"
}
```

**错误码体系**：

| 错误码 | 说明 |
|--------|------|
| 400 | 请求参数错误 |
| 401 | 未授权（token无效或过期） |
| 403 | 权限不足 |
| 404 | 资源不存在 |
| 409 | 资源冲突（重复创建等） |
| 422 | 业务校验失败 |
| 500 | 服务器内部错误 |
| 502 | 上游服务异常（OA接口调用失败） |
| 503 | 服务暂不可用 |

**OA接口专用错误码**：

| 错误码 | 说明 |
|--------|------|
| OA_001 | OA系统版本不存在 |
| OA_002 | OA系统网络超时 |
| OA_003 | OA系统权限不足 |
| OA_004 | OA系统返回数据格式错误 |

#### 异常处理策略

| 场景 | 处理策略 |
|------|----------|
| OA接口调用失败 | 记录失败日志，3次重试后进入人工处理队列 |
| 附件上传超限 | 返回422错误，提示"文件大小超限（最大50MB）" |
| 并发冲突 | 后请求返回409，提示"资源已被其他操作修改，请刷新后重试" |
| 数据删除时有关联 | 软删除（status='deleted'），关联数据置空而非物理删除 |

## 6. 业务流程

### 6.1 项目类测试流程
```
项目创建 → 用例提供 → 评审会议 → 评审通过 → 执行测试 → 缺陷跟踪 → 项目归档
```

### 6.2 日常需求测试流程
```
OA推送版本 → 版本接收 → 分配负责人 → 需求关联 → 编写用例 → 执行测试 → 结果回传OA
```

### 6.3 版本退回流程
```
测试不通过 → 填写退回原因 → 退回版本（自动调用OA接口）→ OA处理 → 再次提测
```
